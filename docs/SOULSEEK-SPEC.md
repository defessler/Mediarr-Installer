# Mediarr Installer — Soulseek (slskd + soularr) Integration Spec

Status: research-complete, verified against upstream docs + the Mediarr codebase at commit `b4fc723`. Every external fact is cited; every assumption is flagged `[ASSUMPTION]`.

## 0. Executive summary

Add two new opt-in containers behind a single `ENABLE_SOULSEEK` flag (default `false`):

- **slskd** (`slskd/slskd:latest`) — the Soulseek daemon. Routed **through the VPN**, mirroring qBittorrent: `network_mode: container:gluetun`, WebUI port published **on gluetun** + opened via gluetun `FIREWALL_INPUT_PORTS`. In the no-vpn override it moves to the `media` bridge with its own published port.
- **soularr** (`mrusse08/soularr:latest`) — the Lidarr↔slskd bridge. Runs on the `media` bridge (it needs DNS to both `lidarr` and the gluetun-published slskd WebUI). Internal daemon loop via `SCRIPT_INTERVAL`. Its `config.ini` is generated by `setup-arr-config.py`.

Both behind a new compose profile `soulseek`. slskd configured purely via `SLSKD_*` env vars; soularr configured via a generated `config.ini`. A single shared downloads path under `DATA_ROOT` keeps slskd, soularr, and Lidarr agreeing on file locations so imports/hardlinks work.

This is **additive + back-compat**: with `ENABLE_SOULSEEK=false` (the default) no new container starts, the `soulseek` profile is never added to `COMPOSE_PROFILES`, and every generated file is byte-identical to today.

### 1.2 soularr (source: README + `config.ini` + `docker-compose.yml` on `main`)

- Image: `mrusse08/soularr:latest` (DockerHub + GHCR).
- Runs as an **internal daemon loop**, interval via env `SCRIPT_INTERVAL` (seconds; example `300`). Also exposes an optional WebUI (`WEBUI_ENABLED`, `WEBUI_PORT=8265`) — **we will leave the WebUI off** to keep the surface minimal `[ASSUMPTION: WebUI is optional; README shows it env-gated]`.
- Volumes: `/data` (config dir, holds `config.ini` + `soularr.log`) and `/downloads` (must equal slskd's downloads dir; Lidarr must see the same files).

## 3. The shared downloads path (the load-bearing decision)

Existing convention (verified): qBit mounts `${DATA_ROOT}/Downloads/Torrents → /downloads` (`docker-compose.yml:539`); arrs/unpackerr mount `${DATA_ROOT} → /data` and reference completed dirs as `/data/Downloads/...` (`setup-arr-config.py:3045`).

We introduce one new host dir: **`${DATA_ROOT}/Downloads/Soulseek`**, mounted into all three relevant containers:

| Container | Host path | Container path | Used as |
|---|---|---|---|
| slskd | `${DATA_ROOT}/Downloads/Soulseek` | `/downloads` | `SLSKD_DOWNLOADS_DIR=/downloads` |
| soularr | `${DATA_ROOT}/Downloads/Soulseek` | `/downloads` | `[Slskd] download_dir = /downloads` |
| lidarr (already mounts `${DATA_ROOT}→/data`) | — | `/data/Downloads/Soulseek` | `[Lidarr] download_dir = /data/Downloads/Soulseek` |

Because Lidarr already bind-mounts the whole `${DATA_ROOT}` as `/data` (`docker-compose.yml:396`), **no Lidarr compose change is needed** — it already sees `/data/Downloads/Soulseek`. This is the path-mapping correctness guarantee: identical physical bytes, hardlink/atomic-move-capable (same filesystem as Lidarr's library under `DATA_ROOT`).

slskd's incomplete dir stays inside `/app` (default `/app/incomplete`) so partial files never appear to Lidarr.

## 4. Concrete changes

### 4.1 docker-compose.yml additions

Add to the gluetun service (`docker-compose.yml`, edit `:436` and `:486`):

```yaml
    environment:
      ...
      # Allow LAN to reach qBittorrent (49156) AND slskd (5030) WebUIs
      # through the VPN container. Comma-separated.
      - FIREWALL_INPUT_PORTS=49156,5030
    ports:
      - ${LAN_IP}:49156:49156
      # slskd WebUI, published on gluetun because slskd shares this
      # namespace (network_mode: container:gluetun). Only bound when the
      # "vpn" profile is active, which only happens when ENABLE_SOULSEEK
      # or ENABLE_QBITTORRENT pulls in gluetun.
      - ${LAN_IP}:5030:5030
```

`[ASSUMPTION]` Adding `5030` to gluetun's published ports unconditionally is safe even when Soulseek is off, because gluetun only starts when the `vpn` profile is active (qBit-enabled). But to keep it clean and avoid binding 5030 for torrent-only users, **prefer** making it a templated/overlaid port — see §6 build note. Simplest correct option that preserves back-compat: leave gluetun's `5030` publish in place; it is harmless when slskd isn't running (nothing answers, but the host bind succeeds). **Verify live** that gluetun doesn't refuse to start when an `INPUT_PORT` has no internal listener (it doesn't — it's a firewall rule, not a proxy).

Add the two new services (place slskd near qBittorrent under "Download Clients", soularr near the arrs):

```yaml
  slskd:
    image: slskd/slskd:latest
    container_name: slskd
    profiles: ["soulseek"]
    # Mirror qBittorrent: live inside gluetun's network namespace so all
    # Soulseek traffic egresses through the VPN. WebUI (5030) is published
    # ON gluetun (see gluetun ports + FIREWALL_INPUT_PORTS=...,5030).
    network_mode: "container:gluetun"
    depends_on:
      - gluetun
    environment:
      - PUID=${PUID}
      - PGID=${PGID}
      - TZ=${TZ}
      - UMASK=002
      # Soulseek network credentials
      - SLSKD_SLSK_USERNAME=${SLSKD_USER}
      - SLSKD_SLSK_PASSWORD=${SLSKD_PASS}
      # WebUI auth stays ENABLED (default user/pass slskd/slskd is replaced
      # by the API-key path for soularr). We do NOT disable auth on a
      # VPN-fronted-but-LAN-published port.
      - SLSKD_API_KEY=${SLSKD_API_KEY}
      # Downloads to the shared host dir; incomplete stays in /app.
      - SLSKD_DOWNLOADS_DIR=/downloads
      # No remote (web) config — config is env-driven + immutable.
      - SLSKD_REMOTE_CONFIGURATION=false
      # Peer listen port. With NordVPN there is NO inbound port-forward,
      # so this is best-effort outbound; search/browse quality is degraded
      # (documented). Kept at the default for when PF-capable providers
      # are used.
      - SLSKD_SLSK_LISTEN_PORT=50300
    volumes:
      - ${INSTALL_DIR}/slskd/config:/app
      - ${DATA_ROOT}/Downloads/Soulseek:/downloads
    logging: *logging
    restart: unless-stopped

  soularr:
    image: mrusse08/soularr:latest
    container_name: soularr
    profiles: ["soulseek"]
    networks: [media]
    security_opt: *security
    # No depends_on lidarr/slskd: lidarr is profile-gated (a hard dep on a
    # profile-excluded service fails compose validation — same rule as
    # bazarr/unpackerr at :353-356), and slskd is in gluetun's namespace
    # (not a "media" sibling). soularr retries over HTTP, so boot order
    # doesn't matter.
    environment:
      - PUID=${PUID}
      - PGID=${PGID}
      - TZ=${TZ}
      - SCRIPT_INTERVAL=${SOULARR_INTERVAL:-300}
      # WebUI off — config is file-driven and the bridge service surfaces
      # nothing the user needs to click.
      - WEBUI_ENABLED=false
    volumes:
      - ${INSTALL_DIR}/soularr/config:/data
      - ${DATA_ROOT}/Downloads/Soulseek:/downloads
    logging: *logging
    restart: unless-stopped
```

Design notes:
- `soularr` is on the `media` bridge so it resolves `lidarr` by DNS. It reaches slskd's WebUI via the host's gluetun-published `LAN_IP:5030` (written into `config.ini`, §4.3) — exactly the `LAN_IP:49156` qBit pattern (`setup-arr-config.py:3606`).
- `slskd` deliberately has **no** `networks:`/`security_opt:` keys — like qBittorrent (`:503-541`), a container in `network_mode: container:gluetun` cannot also declare networks, and `no-new-privileges` is fine but qBit omits `security_opt` here too; match it for consistency.

### 4.2 docker-compose.no-vpn.yml additions

When VPN is off, slskd moves to the bridge with its own published WebUI port (mirror of the qBit override `:17-29`):

```yaml
  slskd:
    network_mode: bridge
    networks:
      - media
    ports:
      - ${LAN_IP}:5030:5030
      - 50300:50300       # Soulseek peer listen (best-effort, like 6881)
    depends_on: !reset null
    # soularr can now reach slskd as http://slskd:5030 on the bridge, but
    # setup-arr-config.py writes the LAN_IP:5030 form unconditionally so
    # one config.ini works in both modes (see §4.3).
```

No soularr entry needed in the override — it is already a `media`-bridge service in both modes.

### 4.3 soularr config.ini generation (setup-arr-config.py) — see §5.

`[Slskd].host_url` is set to **`http://${LAN_IP}:5030`** (not `http://slskd:5030`), because:
- VPN on: slskd's WebUI is published by gluetun at `LAN_IP:5030`; `slskd` is not a DNS name on the `media` bridge.
- VPN off: `LAN_IP:5030` is also published (override above), so the same value works.

This is the documented `QB_HOST = LAN_IP:port` pattern already used for qBittorrent (`setup-arr-config.py:3606,3846`).

### 5.2 New `main()` block (insert in the "Config files" section, right after the Unpackerr block at `:4146`, before Recyclarr)

```python
section("Soularr")
if is_enabled(env, 'ENABLE_SOULSEEK'):
    if not is_enabled(env, 'ENABLE_LIDARR') or not LIDARR_KEY:
        print(f"  {DIM}⏭  Soulseek needs Lidarr — enable Lidarr and re-run.{RESET}")
    else:
        conf_path = f"{B}/soularr/config/config.ini"
        slskd_key = env.get('SLSKD_API_KEY', '')
        new_content = SOULARR_CONF.format(
            lidarr_key=LIDARR_KEY,
            lidarr_int=LIDARR_INT,                       # http://lidarr:8686  (:3611)
            lidarr_dl="/data/Downloads/Soulseek",        # Lidarr's view
            slskd_key=slskd_key or 'REPLACE_WITH_SLSKD_KEY',
            slskd_host=f"http://{LAN_IP}:5030",          # gluetun-published, VPN on+off
        )
        try:
            with open(conf_path, 'r', encoding='utf-8') as f:
                old_content = f.read()
        except FileNotFoundError:
            old_content = ''
        backup_before_overwrite(conf_path)
        overwrite_config_file("Soularr", conf_path, new_content)
        # soularr re-reads config.ini each loop, but bounce it so a fresh
        # key/path takes effect immediately rather than next interval.
        if slskd_key and new_content != old_content:
            _restart_container("soularr")   # mirror unpackerr's restart (:4146+)
else:
    print(f"  {DIM}⏭  ENABLE_SOULSEEK=false — Soulseek not deployed.{RESET}")
```

Notes:
- `LIDARR_KEY`, `LIDARR_INT`, `B`, `LAN_IP` all already exist in `main()` (`:3677,3611,3619,3569`).
- `SLSKD_API_KEY` is generated by the wizard / setup.sh (see §6 build order) and round-trips through `.env`; here we read it from `env`.
- Reuse the same restart helper unpackerr uses (`:4146+`); if that's inline rather than a named function, inline the equivalent `docker restart soularr` guarded by container existence.
- slskd needs **no** API call from this script — it self-configures from env. We do **not** add a `configure_slskd()`; the only programmatic consumer of slskd is soularr.

### 5.3 No change to the key-resolution loop
slskd has no `config.xml` and no discoverable key — its key is **wizard-generated** and authoritative in `.env`. Do not add it to `arr_configs` (`:3645`).


## 9. Risks / unknowns / what to verify live

1. **NordVPN has no port-forwarding (confirmed in-repo `:479-482`, `.env.example:190`).** slskd's inbound listen port (50300) is therefore unreachable from the internet. **Impact (documented):** *poor search results + inability to browse/retrieve user info for some users* — downloads still work. **Verify live:** that real-world download success on NordVPN is acceptable; if not, document "use a PF-capable provider (Proton/PIA) for best Soulseek results," exactly as the repo already advises for qBit seeding.

2. **slskd-in-gluetun wedge** is the *same* immutable-network-mode + stale-namespace problem qBit has (`setup.sh:538-570,1052-1063`, plus the `qbit-guardian.sh` cron `:1152`). **Must mirror all four fixes** for slskd (VPN-toggle reconcile, post-up stale-namespace recreate, guardian cron, restart policy). **Verify live:** toggle `VPN_ENABLED` true→false→true and recreate gluetun; confirm slskd re-welds and its WebUI returns on 5030.

3. **Soularr scheduling:** internal loop (`SCRIPT_INTERVAL`), not one-shot — **no host cron needed** (verified from README). Default 300s. **Verify live:** that the loop honors a config.ini written *after* container start (we restart it to be safe, mirroring unpackerr `:4146`).

4. **Path-mapping (the classic soularr failure):** mitigated by the single shared host dir `${DATA_ROOT}/Downloads/Soulseek` (§3) and PUID/PGID parity (all three containers use `${PUID}/${PGID}`, matching slskd docker docs' `user: 1000:1000` guidance and soularr's "lidarr and slskd must agree on user/group"). **Verify live:** download a FLAC via soularr, confirm Lidarr imports (hardlink, not copy) from `/data/Downloads/Soulseek`.

5. **slskd host_url for soularr = `LAN_IP:5030`, not `slskd:5030`:** because slskd is in gluetun's namespace, the compose-DNS form from the upstream sample **will not resolve**. This is the qBit `LAN_IP:49156` pattern (`setup-arr-config.py:3606`). **Verify live:** `docker exec soularr wget -qO- http://$LAN_IP:5030` returns the slskd UI; and confirm gluetun's `FIREWALL_INPUT_PORTS=...,5030` actually admits the LAN→gluetun→slskd path (the `FIREWALL_OUTBOUND_SUBNETS` reply-routing fix at `:437-445` is what makes sibling→gluetun-published-port work — same dependency qBit relies on).

6. **Share-to-download consent:** Soulseek is a sharing community. We ship `SLSKD_SHARED_DIR` **unset** (no shares) by default to avoid surprising users into exposing files. **Unknown/verify:** whether downloads succeed acceptably while sharing nothing (server-side leech policies are undocumented in slskd). Surface a Configure-screen toggle "Share my music library back to Soulseek (recommended; improves results)" that, when on, sets `SLSKD_SHARED_DIR=/data/Media/Music` (read-only mount `[ASSUMPTION on music library path — confirm against the user's Lidarr root folder]`). Default off + explicit consent.

7. **`SLSKD_API_KEY` generation:** verified that the env var alone mints a working Administrator key (CIDR `0.0.0.0/0`). **Decide & verify:** generate it in the wizard (preferred — unique per install) vs. in setup.sh (`openssl rand -hex 24`) if the user left it blank. Whichever, write it back to `.env` so soularr's `config.ini` and slskd's env agree. **Verify live:** `curl -H "X-API-Key: $KEY" http://$LAN_IP:5030/api/v0/session` succeeds while WebUI auth remains enabled.

8. **gluetun port 5030 published unconditionally:** when only qBit is enabled (Soulseek off), gluetun still binds `LAN_IP:5030` with nothing behind it. Harmless but untidy. **Verify live** gluetun starts fine; if it complains, gate the 5030 publish behind an override file added only when `ENABLE_SOULSEEK=true` (cleaner but more plumbing) — see build order step 2.

9. **`dist`/`resources` copies:** `installer/dist/...` and `installer/resources/nas-payload/scripts/...` are build outputs of `nas/scripts/...`. Edit **only** `nas/scripts/*` and `installer/src/*`; the build step repopulates the copies. **Verify** the build wiring (likely an electron-builder `extraResources` copy) actually syncs `nas/scripts` → `installer/resources/nas-payload/scripts`.

10. **`security_opt`/`networks` on slskd:** omitted to match qBit's namespace-shared form. **Verify** compose validation passes (declaring `networks:` alongside `network_mode: container:` is an error).
